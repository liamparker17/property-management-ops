## How to use this document

Each task below is **self-contained**. A fresh Claude instance should be able to pick one up with no prior conversation and execute it. Every task includes:

- **ID** — stable handle (e.g. `T-M2-01`).
- **Depends on** — tasks that must land first.
- **Read first** — exact file paths and plan sections to load (check `CODEBASE.md` before opening code).
- **Deliverables** — files to create/edit, with pointer back to the plan section that defines the shape.
- **Verification** — commands to run; what "done" looks like.
- **Manifest update** — which table in `CODEBASE.md` to update.
- **Clarifications needed** — resolve with the user before touching code. "None" means ready to execute.

Binding constraints that apply to **every task**:

- Read `CODEBASE.md` before any source file (manifest-first rule).
- Business logic in `lib/services/*`. Validation in `lib/zod/*`. Routes are thin `withOrg()` wrappers.
- Money is `Int` cents, formatted via `formatZar()` in `lib/format.ts`.
- UI follows the Regalis shell invariants in plan §0.1. Do not introduce new card/header/badge primitives.
- Every nav entry must have a destination page in the same PR. No dead links.
- Update `CODEBASE.md` in the same PR as the change.
- Every migration: `npx prisma migrate dev --name <snake_case_name>` and commit the generated SQL.
- Secrets and tokens never land in the repo. New integration credentials route through `OrgIntegration` (Phase A).
- If anything in the clarifications list is unanswered, **stop and ask** — do not guess.

## Locked decisions for M2 (answered 2026-04-24)

These are binding for every M2 task. If a task description contradicts a locked decision, the decision wins.

1. **Trust accounts are per-landlord, not per-org.** Change `TrustAccount.orgId @unique` to composite `@@unique([orgId, landlordId])`; drop the per-org singleton. Every `TrustLedgerEntry` MUST carry both `trustAccountId` (resolving to the right landlord) and `landlordId`. `ensureTrustAccount(orgId, landlordId)` replaces `ensureTrustAccount(orgId)`.

2. **Payment rails — four rails, all stubbed** (no real integration keys yet):
   - **Rail A — Stitch inbound**: per-org white-label Stitch Connect OAuth. Tenant portal "Pay now" button → hosted Stitch checkout → webhook → `PaymentReceipt { source: STITCH }`.
   - **Rail B(i) — DebiCheck via Stitch**: per-org Stitch DebiCheck product. Tenant signs mandate in their banking app, we store mandate id, we submit monthly collections against the mandate, upper-threshold cap honored, auto-retry once at +2 business days on unpaid, then flip invoice to OVERDUE and notify.
   - **Rail B(ii) — Self-managed tenant debit order**: tenant sets up debit order at their own bank. We generate a one-page instruction sheet PDF (our bank + account + branch, unique reference = lease id, suggested cap = rent + 25%). Incoming money reconciles by reference. No retry, no mandate control. Tenant portal shows "Debit order active (self-managed)".
   - **Rail C — Stitch outbound**: per-org Stitch payouts. Landlord payout flow: PM initiates disbursement → Stitch payout → `TrustLedgerEntry { type: DISBURSEMENT }`.
   - **Rail D — Reconciliation**: twice-daily pull (06:00 + 18:00 SAST) from **QuickBooks Online** per-org connected realm (preferred) or manual CSV upload (fallback / pre-QBO). Matching engine pairs bank-feed transactions to `PaymentReceipt` by reference/amount/date. Unmatched rows become `ReconciliationException { kind: UNMATCHED_BANK_TX }`.

3. **Shared credential storage — `OrgIntegration` model (NEW).** Per-org, per-provider encrypted tokens. All integrations (Stitch, QBO, TPN) route through it. TPN stub from M1 must be refactored onto this model. Shape:
   ```prisma
   enum IntegrationProvider { STITCH_PAYMENTS STITCH_DEBICHECK STITCH_PAYOUTS QUICKBOOKS TPN }
   enum IntegrationStatus   { DISCONNECTED CONNECTED ERROR }
   model OrgIntegration {
     id                 String @id @default(cuid())
     orgId              String
     provider           IntegrationProvider
     status             IntegrationStatus @default(DISCONNECTED)
     externalAccountId  String?
     accessTokenCipher  String?
     refreshTokenCipher String?
     tokenExpiresAt     DateTime?
     connectedAt        DateTime?
     connectedById      String?
     lastError          String?
     updatedAt          DateTime @updatedAt
     org                Org @relation(fields: [orgId], references: [id], onDelete: Cascade)
     @@unique([orgId, provider])
   }
   ```
   Add `lib/crypto.ts` with `encrypt`/`decrypt` using `INTEGRATION_SECRET_KEY` env var (32-byte hex, AES-256-GCM).

4. **Per-org Stitch liability.** Org signs up to Stitch via white-label partner flow; org is liable for fees/KYC. We never hold card/bank PANs; Stitch tokenises. App opens Stitch's partner-branded onboarding in a modal; callback persists `OrgIntegration`.

5. **QBO onboarding.** If org already has QBO → standard OAuth. If not → "Sign up for QuickBooks" affiliate/referral link in the connect modal that takes them to QBO signup with our partner id, then returns to the OAuth flow. Mark the distinction in analytics but same code path.

6. **Defaults accepted** for the other M2 clarifications:
   - `Invoice.amountCents` kept as cache indefinitely (drop in follow-up only if drift occurs).
   - Line-item backfill runs live (idempotent).
   - Meters per-unit only (shared meters are future work).
   - `UtilityTariff` org-wide default with optional `propertyId` override (schema already supports this).
   - Statements on-demand only for M2 (auto cadence lands in M4).
   - Partial payments: auto-allocate oldest-first with manual re-allocate UI.
   - Reversal window: 30 days for non-ADMIN, indefinite for ADMIN (audit-logged).
   - PDF regeneration allowed after period lock; ledger snapshot is frozen.

7. **Approvals stay plan-aligned** (single-stage by `Org.ownerType`). Do not change approval routing in M2.

8. **Acceptance gate reuses T-P1-18 pattern.** M2 closes when typecheck + tests + nav validator + migrate status all pass.
