## How to use this document

Each task below is **self-contained**. A fresh Claude instance should be able to pick one up with no prior conversation and execute it. Every task includes:

- **ID** — stable handle (e.g. `T-M4-01`).
- **Depends on** — tasks that must land first.
- **Read first** — exact file paths and plan sections to load (check `CODEBASE.md` before opening code).
- **Deliverables** — files to create/edit, with pointer back to the plan section that defines the shape.
- **Verification** — commands to run; what "done" looks like.
- **Manifest update** — which table in `CODEBASE.md` to update.
- **Clarifications needed** — resolve with the user before touching code. "None" means ready to execute.

Binding constraints that apply to **every task**:

- Read `CODEBASE.md` before any source file (manifest-first rule).
- Business logic in `lib/services/*`. Validation in `lib/zod/*`. Routes are thin `withOrg()` wrappers.
- Money is `Int` cents, formatted via `formatZar()` in `lib/format.ts`.
- UI follows the Regalis shell invariants in plan §0.1. Do not introduce new card/header/badge primitives. Analytics primitives must satisfy plan §0.1.3.
- Every nav entry must have a destination page in the same PR. No dead links.
- Update `CODEBASE.md` in the same PR as the change.
- Every migration: `npx prisma migrate dev --name <snake_case_name>` and commit the generated SQL.
- Secrets and tokens never land in the repo. New integration credentials route through `OrgIntegration` (landed in M2).
- Every new analytics query is `orgId`-scoped via `ctx` and additionally role-scoped via `withRoleScopeFilter` (T-M4-07).
- All third-party integrations remain stub-mode (locked decision #11). No live network calls in CI or dev.
- If anything in the clarifications list is unanswered, **stop and ask** — do not guess.

## Locked decisions for M4 (answered 2026-04-26)

These are binding for every M4 task. If a task description contradicts a locked decision, the decision wins.

1. **Snapshot refresh is real-time on mutation, not cron-batched.** Every service that mutates a tracked metric (lease state, payment receipt, allocation, ledger entry, maintenance status, inspection completion, deposit settlement) calls `recordSnapshotEvent(ctx, kind)` which upserts the relevant `*MonthlySnapshot` rows for the affected period. No `analytics-refresh` cron job. Hourly batch refresh listed in the original plan §8 is dropped.

2. **Map library: `react-leaflet` + OpenStreetMap.** Add `react-leaflet` and `leaflet` to dependencies. Use OSM tiles. No paid Mapbox token. Property pin markers with light styling (Regalis muted palette). Map components live under `components/analytics/maps/`.

3. **Chart library: `recharts`.** Add to dependencies. All chart primitives live under `components/analytics/charts/` and accept a single `data` prop + a registry-derived theme. No other charting libraries.

4. **Role data scoping** — all queries org-scoped via `ctx.orgId`. Additionally:
   - Landlord users (`role === 'LANDLORD'`) see only data for properties where `Property.landlordId === ctx.user.landlordId`.
   - Managing-agent users (`role === 'MANAGING_AGENT'`) see only data for properties where `Property.assignedAgentId === ctx.user.managingAgentId`.
   - Tenant users see only their own lease/invoice/inspection/notification rows.
   - Service helpers add a `withRoleScopeFilter(ctx, propertyClause)` shim used by every analytics + notice query.

5. **Area notice audience is dynamic — resolved at delivery.** `AreaNotice` defines an `audienceQuery` (Json shape: `{ propertyIds?, unitTypes?, leaseStates?, roles? }`). Each delivery attempt (email send, SMS send, in-app fetch) re-resolves recipients via `resolveNoticeAudience(ctx, notice)` so new tenants get caught and ex-tenants drop off automatically. `NoticeDelivery` rows are created at send time, not at notice creation.

6. **Payment alert cadence: 7 days before due → reminder; 14 days past due → overdue; 30 days past due → final.** Cron-driven. Idempotent: each alert kind written at most once per `(invoiceId, kind)`.

7. **Usage alert basis: rolling 3-month per-meter average.** Computed each cron tick: for every active meter, compare the latest `MeterReading` delta against the trailing 3-month mean delta. Threshold defaults: warn at +50%, urgent at +100% (configurable via `OrgFeature.USAGE_ALERTS.config`).

8. **Notification channels** — `Notification` (in-app) is mandatory and instant. `NotificationDelivery` rows model the email + SMS attempts. Email is on by default per org; SMS opt-in via `OrgFeature.PAYMENT_ALERTS.config.smsEnabled` etc. Email sender uses existing `lib/email.ts::sendEmail`; SMS uses existing `lib/sms.ts`.

9. **Outages — hybrid PM + EskomSePush.**
   - `LoadSheddingOutage` model is its own thing (separate from `AreaNotice`). Fields: `orgId, propertyId?, eskomAreaCode?, source ('PM' | 'ESKOM_SE_PUSH'), startsAt, endsAt, stage?, note?, externalEventId?`.
   - PM-created outages have `source: PM`. They're displayed in tenant + landlord portals.
   - A daily cron (`/api/cron/eskom-sync`) polls EskomSePush per-org via `OrgIntegration { provider: ESKOM_SE_PUSH }`. For each `Property.eskomAreaCode` (or auto-resolved from `Property.suburb` + `province` when `eskomAreaCode` is null), fetch the next 48hrs of events. For each event, if no `LoadSheddingOutage` exists with matching `(propertyId, externalEventId)`, insert with `source: ESKOM_SE_PUSH`.
   - EskomSePush is a **stubbed integration** in M4 — adapter at `lib/integrations/eskom/adapter.ts`, gated on `OrgIntegration` per locked decision #11. Cron noops when no org has it connected.
   - Add `Property.eskomAreaCode String?` field. Add `IntegrationProvider.ESKOM_SE_PUSH` enum value (extend M2's existing enum).

10. **Cron platform: Vercel Cron** (matches M2/M3 idiom). All new crons added to `vercel.json`. Schedules listed in each task.

11. **All third-party integrations remain stubbed.** Stitch, QBO, TPN (M1/M2) and EskomSePush (M4) all gated behind `OrgIntegration { status: CONNECTED }` rows + adapter stubs. Real credentials are deferred to a later "credentials & launch" milestone post-M5.

12. **WhatsApp Business contractor cascade is deferred to M5+.** No `lib/integrations/whatsapp/*` in M4.

13. **Acceptance gate** mirrors M1/M2/M3: typecheck + tests + nav validator + migrate status all green; `CODEBASE.md` refreshed.

## Environment variables added in M4

- `ESKOM_SE_PUSH_API_BASE` — base URL for EskomSePush HTTP API. Defaults to `https://developer.sepush.co.za/business/2.0` when unset. Per-org token is read from the `OrgIntegration { provider: ESKOM_SE_PUSH }.credentials` JSON, **not** from env.
- `CRON_SECRET` — already exists from M2; reused by the new `/api/cron/*` routes added in this milestone for HMAC-style header guarding.

No other new env vars. Email/SMS credentials are reused from M0/M1.

### T-M4-02 · `recordSnapshotEvent` helper

**Depends on:** T-M4-01
**Read first:** locked decision #1; `lib/services/leases.ts`, `lib/services/payments.ts`, `lib/services/trust.ts`, `lib/services/maintenance.ts`, `lib/services/inspections.ts`, `lib/services/offboarding.ts` — all M2/M3 mutators
**Deliverables:**
- `lib/services/snapshots.ts` exporting:
  - `type SnapshotEventKind = 'LEASE_STATE' | 'PAYMENT' | 'ALLOCATION' | 'LEDGER' | 'MAINTENANCE' | 'INSPECTION' | 'OFFBOARDING'`
  - `recordSnapshotEvent(ctx, kind: SnapshotEventKind, refs?: { propertyId?: string; landlordId?: string; agentId?: string }): Promise<void>` — synchronous best-effort upsert wrapped in `Promise.resolve().catch(reportSnapshotFailure)` so a snapshot write never blocks a primary mutation.
  - `recomputeOrgSnapshot(ctx, periodStart: Date): Promise<OrgMonthlySnapshot>` — full recompute for a period; called by `recordSnapshotEvent` for the `OrgMonthlySnapshot` row.
  - `recomputePropertySnapshot(ctx, propertyId, periodStart): Promise<PropertyMonthlySnapshot>`
  - `recomputeLandlordSnapshot(ctx, landlordId, periodStart): Promise<LandlordMonthlySnapshot>`
  - `recomputeAgentSnapshot(ctx, agentId, periodStart): Promise<AgentMonthlySnapshot>`
  - `monthFloor(d: Date): Date` — UTC first-of-month at 00:00.
- Routing rules inside `recordSnapshotEvent` (which slice each kind touches):
  - `LEASE_STATE` → org + property + landlord + agent.
  - `PAYMENT` → org + landlord.
  - `ALLOCATION` → org + landlord.
  - `LEDGER` → org + landlord.
  - `MAINTENANCE` → org + property + agent.
  - `INSPECTION` → org + property + agent.
  - `OFFBOARDING` → org + landlord + property.
**Verification:** `tests/services/snapshots.test.ts` — assert each kind upserts only the expected slices; assert failures inside `recomputeXxx` do not throw out of `recordSnapshotEvent`.
**Manifest update:** services table — new `lib/services/snapshots.ts`.
**Clarifications needed:** None.
