## How to use this document

Each task below is **self-contained**. A fresh Claude instance should be able to pick one up with no prior conversation and execute it. Every task includes:

- **ID** — stable handle (e.g. `T-M5-01`).
- **Depends on** — tasks that must land first.
- **Read first** — exact file paths and plan sections to load (check `CODEBASE.md` before opening code).
- **Deliverables** — files to create/edit, with pointer back to the plan section that defines the shape.
- **Verification** — commands to run; what "done" looks like.
- **Manifest update** — which table in `CODEBASE.md` to update.
- **Clarifications needed** — resolve with the user before touching code. "None" means ready to execute.

Binding constraints that apply to **every task**:

- Read `CODEBASE.md` before any source file (manifest-first rule).
- Business logic in `lib/services/*`. Validation in `lib/zod/*`. Routes are thin `withOrg()` wrappers.
- Money is `Int` cents, formatted via `formatZar()` in `lib/format.ts`.
- UI follows the Regalis shell invariants in plan §0.1. Do not introduce new card/header/badge primitives. Reports surfaces reuse the M2 statement primitives.
- Every nav entry must have a destination page in the same PR. No dead links.
- Update `CODEBASE.md` in the same PR as the change.
- Every migration: `npx prisma migrate dev --name <snake_case_name>` and commit the generated SQL.
- Secrets and tokens never land in the repo. Backup credentials route through env (no `OrgIntegration` row needed — backups are platform-wide, not per-org-connected).
- Every new analytics/report query is `orgId`-scoped via `ctx` and additionally role-scoped via `withRoleScopeFilter` (M4 helper). Tax packs additionally enforce subject ownership (a landlord sees only their own pack; a tenant only theirs).
- All third-party integrations remain stubbed (locked decision #10). No live SARS, Stitch, QBO, TPN, or EskomSePush calls in CI or dev.
- If anything in the clarifications list is unanswered, **stop and ask** — do not guess.

## Locked decisions for M5 (answered 2026-04-27)

These are binding for every M5 task. If a task description contradicts a locked decision, the decision wins.

1. **Tax packs are accountant-ready (PDF + CSV), not statutory filing.** Keep `TaxPack` as the canonical record; rendering (PDF/CSV) and transmission (future SARS eFiling) are separate concerns. The pack builder MUST accept a pluggable `transmissionAdapter` parameter that defaults to a `recordOnly` no-op adapter; a future `sarsAdapter.submit(pack)` slot is wired but not implemented in M5. `lib/services/tax-reporting.ts` exposes `registerTransmissionAdapter(name, adapter)` so the SARS adapter can be added later without touching pack generation.

2. **Financial year start: March 1 hardcoded.** Per-org configurability deferred. Add a single `FINANCIAL_YEAR_START` constant exported from `lib/format.ts` (or a new `lib/financial-year.ts`) returning `{ month: 3, day: 1 }`. All year-resolution helpers (`resolveFinancialYearForDate(date)`, `currentFinancialYear()`, `previousFinancialYear()`) read from this constant.

3. **Locked-year regeneration: PDFs regenerable, line items immutable.** Once `FinancialYear.lockedAt` is set, `TaxPackLine` rows MUST NOT change. `regenerateTaxPackPdf(packId)` is permitted and writes a fresh PDF buffer to Blob with a new `storageKey`; previous storage keys remain accessible via `TaxPack.previousStorageKeys[]` (Json array — newest first). Generation rerun for the same `(orgId, subjectType, subjectId, yearId)` is rejected as `ApiError.conflict("Tax pack already generated; use regenerate")` once `TaxPack.generatedAt` is set on a locked year.

4. **Backup location: Vercel Blob.** External S3/GCS deferred. Backups land at `backups/<orgId>/<yyyymmdd>/<pg_dump|blob_manifest>.<ext>` with `addRandomSuffix: false` so the path is predictable for verification.

5. **DB backup cadence: daily `pg_dump` + Neon PITR.** Both run. `pg_dump` is the explicit redundancy: even if Neon is fully down we can restore from Vercel Blob. Cron at 02:00 SAST daily (avoids overlap with M2's debicheck-retry at 02:00; schedule the backup at 02:30 SAST = 00:30 UTC).

6. **Encryption disclosure copy (verbatim, approved):**
   > "Data is encrypted at rest by Neon (Postgres) and Vercel Blob (file storage) using provider-managed keys."

   Rendered on `/settings/backup` admin page only. Plain text; no marketing framing.

7. **Backup verification cadence: weekly Sunday 03:00 SAST.** Pulls the most recent successful `BackupSnapshot`, restores to a temporary throwaway Neon branch (or runs `pg_restore --list` for schema-only verification when no scratch DB is connected), checksums file count + line count, writes a `BackupVerificationRun { ok: boolean; details: Json }` row.

8. **Backup retention: 2 years (730 days).** Daily snapshots beyond 730 days are pruned by the same nightly cron (after a successful new snapshot completes). Weekly verification runs older than 730 days also pruned.

9. **Annual pack evidence depth:** include the full audit chain — every `PaymentReceipt`, matched `BankTransaction` (where reconciled), `Allocation`, and `TrustLedgerEntry` for the period — embedded as supporting line groups in the pack. Accountants get the receipts, not just totals.

10. **All third-party integrations remain stubbed** (Stitch payments / DebiCheck / payouts, QBO, TPN, EskomSePush) — defers to a "credentials & launch" milestone post-M5. WhatsApp Business contractor cascade also deferred. Per-org dedicated DB tier (mentioned during M5 planning) is a separate future milestone, not in M5.

11. **Acceptance gate** mirrors prior milestones: typecheck + tests + nav validator + migrate status all green; `CODEBASE.md` refreshed.

## Environment variables added in M5

- `BACKUP_VERIFICATION_NEON_BRANCH_URL` — optional Neon connection string pointing at a throwaway branch used by `runVerification` to perform a real `pg_restore` smoke test. When unset, verification falls back to `pg_restore --list` schema-only validation against the dump artifact and still records a `BackupVerificationRun`.
- `PG_DUMP_BIN` — optional override for the `pg_dump` binary path; defaults to `pg_dump` on `$PATH`. Used by `lib/services/backup.ts::runDailyBackup`.
- `BACKUP_BLOB_PREFIX` — optional path prefix override; defaults to `backups/`. Locked decision #4 relies on the default for path predictability.
- `CRON_SECRET` — already exists from M2/M4; reused by the new `/api/cron/backup-*` routes.
- `BLOB_READ_WRITE_TOKEN` — already exists; reused for backup writes (locked decision #4).

No other new env vars. No new third-party credentials are introduced in M5 (locked decision #10).

### T-M5-05 · `lib/services/tax-reporting.ts` with pluggable transmission adapter
