# Rotating Environment Tokens — End-to-End Runbook

**Scope:** Every secret used by this app, in the order you should rotate them. Covers local (`.env.local`), Vercel (Production + Preview + Development), and the upstream provider where the secret is minted.

**Why rotate:** Suspected leak, a contractor offboarded, scheduled quarterly hygiene, or a secret accidentally committed to git.

**Audience:** You, operating solo on your own machine.

---

## Before you start — the golden rules

1. **Never commit `.env.local` or `.env.production.local`.** They are gitignored. If you ever accidentally commit one, assume every secret in it is leaked and rotate everything in this doc.
2. **Rotate in the order below.** `NEXTAUTH_SECRET` last — rotating it signs every logged-in user out instantly, so do it after everything else is confirmed working.
3. **Update Vercel first, then redeploy, then update local.** This avoids a window where Vercel still holds the old secret while the upstream provider is expecting the new one.
4. **Keep the old value open in one tab** until the new one is confirmed working end-to-end. If the rotation fails, paste the old value back while you debug.
5. **After each rotation, spot-check the feature that uses the secret.** A successful deploy doesn't prove the secret works — only a real login / email / upload / SMS does.

---

## Quick reference — every secret in this app

| # | Variable | Provider | Blast radius if leaked |
|---|---|---|---|
| 1 | `DATABASE_URL` | Neon | Full read/write access to all PM + tenant data |
| 2 | `DIRECT_URL` | Neon | Same as DATABASE_URL (schema migrations) |
| 3 | `BLOB_READ_WRITE_TOKEN` | Vercel Blob | Read/write on all uploaded docs (leases, IDs) |
| 4 | `GMAIL_APP_PASSWORD` | Google | Can send email from your Gmail account |
| 5 | `SMS_GATEWAY_PASSWORD` | SMS Gateway for Android | Can send SMS from your Android phone (billed to you) |
| 6 | `SENTRY_DSN` | Sentry | Can inject fake errors into your dashboard (low risk) |
| 7 | `NEXTAUTH_SECRET` | Generated by you | Signs JWTs — rotate **last**, logs everyone out |

Non-secret config values (`NEXTAUTH_URL`, `APP_URL`, `EMAIL_FROM*`, `EXPIRING_WINDOW_DAYS`, `OPS_SMS_RECIPIENTS`, `GMAIL_USER`, `SMS_GATEWAY_USER`) don't need rotation — they're identifiers, not credentials.

## 1. `DATABASE_URL` + `DIRECT_URL` (Neon)

Both point at the same Neon Postgres; `DATABASE_URL` is the pooled connection and `DIRECT_URL` is the unpooled one used by Prisma for migrations. They rotate together.

### Steps
1. Open [console.neon.tech](https://console.neon.tech) → your project → **Connection Details**.
2. Click the password field → **Reset password**. Neon will mint a new password on the existing role.
3. Copy the new **pooled** and **unpooled** connection strings (two separate strings — Neon shows a toggle).
4. On Vercel: project → **Settings → Environment Variables**.
   - Edit `DATABASE_URL` — paste pooled. Apply to Production, Preview, Development.
   - Edit `DIRECT_URL` — paste unpooled. Apply to all three environments.
5. Redeploy: **Deployments → latest → Redeploy** (check "use existing build cache" — ok here).
6. Update `.env.local`:
   ```
   DATABASE_URL="<new pooled>"
   DIRECT_URL="<new unpooled>"
   ```
7. Verify locally: `npx prisma db pull` — should connect and output "Introspected X models".
8. Verify prod: log into the deployed app → list properties → confirm data loads.

### What breaks if you skip a step
Vercel holds the old password → all prod queries fail with `password authentication failed`. Roll back by pasting the old string back into Neon's password reset, or resetting again on Neon.

## 2. `BLOB_READ_WRITE_TOKEN` (Vercel Blob)

### Steps
1. Vercel dashboard → **Storage** → select your Blob store → **Settings** tab.
2. Under **Tokens**, click **Create Token** → name it something like `pmops-prod-2026-Q2`. Copy the token value — it's only shown once.
3. Back on Vercel: project → **Settings → Environment Variables** → edit `BLOB_READ_WRITE_TOKEN`. Paste the new value, apply to Production, Preview, Development.
4. Redeploy.
5. Update `.env.local` with the new token.
6. Verify: in the app, upload a lease PDF via a lease's documents tab → confirm it uploads and the download link works.
7. **After confirmed working**, go back to the Blob store → **Tokens** → revoke the old token. Don't skip this — the old token still works until revoked.

### What breaks if you skip a step
If you revoke the old token before the new one is deployed, every upload in prod 500s until redeploy finishes.

## 3. `GMAIL_APP_PASSWORD` (Google)

App Passwords are per-account, 16-character credentials. The Gmail address in `GMAIL_USER` stays the same.

### Steps
1. Go to [myaccount.google.com/apppasswords](https://myaccount.google.com/apppasswords). Requires 2-Step Verification on the account.
2. Under **Your app passwords**, find the existing PMOps entry (if you named one) and click the trash icon — **do not yet delete it**, just note it.
3. Click **Create new app password** → name it `pmops-<date>` → **Create**.
4. Copy the 16-character password (no spaces needed).
5. Vercel env vars → edit `GMAIL_APP_PASSWORD` → paste, apply to all three envs → Save.
6. Redeploy.
7. Update `.env.local`.
8. Verify: create a new tenant via `/tenants/onboard` with "Send invite" toggled on. Confirm the invite email arrives.
9. Once confirmed, delete the old app password on the Google page.

### What breaks if you skip a step
Google silently accepts the new password immediately, but old cached SMTP sessions can keep working for a few minutes. If email fails right after rotation, wait 2 minutes and retry before panicking.

---

## 4. `SMS_GATEWAY_PASSWORD` (SMS Gateway for Android)

The gateway password is set inside the Android app on your phone.

### Steps
1. Open the **SMS Gateway** app on your Android phone.
2. **Settings → Cloud Server** (the public cloud mode — confirm "Cloud Server" is toggled on).
3. Tap the **Password** field → generate new → copy it. Username usually stays the same unless you explicitly change it.
4. Vercel env vars → edit `SMS_GATEWAY_PASSWORD` → paste, apply to all three envs → Save.
5. Redeploy.
6. Update `.env.local`.
7. Verify: trigger an SMS — easiest way is to onboard a tenant with a phone number (the invite SMS uses the gateway). Check your Android phone's sent SMS folder to confirm the message went out.

### What breaks if you skip a step
If the phone app password diverges from the env var, API calls return 401 and SMS silently no-ops (per `lib/sms.ts` design — ops alerts just stop arriving, which is easy to miss).

---

## 5. `SENTRY_DSN` (optional)

DSNs are public-ish (embedded in client bundles if used client-side). Rotation is low-value unless you're cleaning up old projects.

### Steps
1. [sentry.io](https://sentry.io) → your project → **Settings → Client Keys (DSN)**.
2. Click **Generate New Key**. New DSN is active immediately.
3. Update Vercel env var → redeploy.
4. Update `.env.local`.
5. Once confirmed errors still flow to Sentry, revoke the old DSN on the same page.

---

## 6. `NEXTAUTH_SECRET` — do this **last**

Used to sign the session JWT. Rotating it invalidates every existing session — all logged-in users (PMs, tenants, landlords, agents) will be signed out and need to log in again.

### Steps
1. Generate a new secret locally:
   ```bash
   openssl rand -base64 32
   ```
2. On Vercel: project → **Settings → Environment Variables** → edit `NEXTAUTH_SECRET` → paste the new value → apply to Production, Preview, Development → Save.
3. Redeploy.
4. Update `.env.local` with the same new value.
5. Verify: log out, then log back in on prod. If login works and protected pages load, rotation succeeded.

### What breaks if you skip a step
- **Different secrets between local and Vercel** is fine — they're separate deployments. Problem is only if local dev tries to validate a JWT signed on prod (it won't normally; each env signs its own).
- **Mid-rotation window:** between pasting the new secret and redeploy finishing, Vercel may be serving requests with cached env. Users hitting protected pages during that window see 401s. Do this during low traffic.

## When the rotation is because a secret leaked

If the reason for rotating is that a secret got into git, a log, or a screenshot:

1. **Rotate immediately** — don't wait for a "good time" even if it signs everyone out.
2. **Check git history**: `git log -p | grep <leaked-value>` — if it's in history, the repo must be treated as compromised. Consider scrubbing history with `git filter-repo` or (last resort) rewriting the branch.
3. **Check Vercel deployment logs** for the time window between the leak and the rotation — look for anomalous database or blob activity.
4. **Review Neon audit log** at [console.neon.tech](https://console.neon.tech) → Project → Operations.
5. **Check Gmail sent folder** for emails you didn't send.
6. **After the rotation is confirmed working**, consider whether to tell anyone (tenants, landlords, co-owners) — depends on what the secret exposed. POPIA may require notification if personal data was accessible.

## Checklist — printable version

- [ ] `.env.local` backed up to a password manager (not committed to git)
- [ ] `DATABASE_URL` rotated on Neon, Vercel, `.env.local` — verified via `prisma db pull`
- [ ] `DIRECT_URL` rotated on Neon, Vercel, `.env.local`
- [ ] `BLOB_READ_WRITE_TOKEN` rotated on Vercel Blob, Vercel env, `.env.local` — verified via real upload
- [ ] `GMAIL_APP_PASSWORD` rotated on Google, Vercel, `.env.local` — verified via real tenant invite
- [ ] `SMS_GATEWAY_PASSWORD` rotated on Android app, Vercel, `.env.local` — verified via real SMS send
- [ ] `SENTRY_DSN` rotated (if applicable)
- [ ] Old Blob token revoked
- [ ] Old Gmail app password deleted
- [ ] Old Sentry DSN revoked
- [ ] `NEXTAUTH_SECRET` rotated **last** — verified by logging out and back in

## Frequency recommendations

| Secret | Rotate how often |
|---|---|
| `NEXTAUTH_SECRET` | Every 6–12 months, or on compromise |
| `DATABASE_URL` / `DIRECT_URL` | Every 12 months, or on compromise / contractor offboarding |
| `BLOB_READ_WRITE_TOKEN` | Every 12 months |
| `GMAIL_APP_PASSWORD` | Every 6 months (Google sometimes invalidates dormant ones anyway) |
| `SMS_GATEWAY_PASSWORD` | On device change or compromise |
| `SENTRY_DSN` | Only on compromise |
